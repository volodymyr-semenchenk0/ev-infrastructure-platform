# 3. РЕАЛІЗАЦІЯ СИСТЕМИ ВИБОРУ ЛОКАЦІЙ ЗАРЯДНИХ СТАНЦІЙ

## 3.1. Програмна реалізація системи вибору локацій зарядних станцій

Спроектована у розділі 2 система втілюється у програмний продукт, що відтворює гібридну процедуру FAHP–TOPSIS–Монте-Карло як послідовність кроків оцінювання локацій-кандидатів. Реалізація охоплює серверне обчислювальне ядро, клієнтський інтерфейс багатокрокового сеансу та форми подання ранжування і його стійкості.

### 3.1.1. Програмні засоби та реалізація функцій системи

Серверну частину разом із математичним ядром написано мовою Python, клієнтську частину – мовою TypeScript. Розподіл мов відповідає трирівневій клієнт-серверній архітектурі, обґрунтованій у попередньому розділі. Серверна і клієнтська частини взаємодіють лише через REST API, специфікацію якого описано форматом OpenAPI. Перелік мов, фреймворків, бібліотек і допоміжних засобів, застосованих для реалізації функцій системи, наведено у табл. 3.1.

Таблиця 3.1. – Програмні засоби реалізації системи вибору локацій зарядних станцій

| Рівень | Технологія | Призначення |
|---|---|---|
| Серверна частина | FastAPI | Асинхронний веб-фреймворк, генерація специфікації OpenAPI |
| Серверна частина | SQLAlchemy + GeoAlchemy2 | Асинхронний ORM, відображення просторових типів PostGIS |
| Математичне ядро | NumPy, SciPy | Векторні обчислення, статистичні функції |
| База даних | PostgreSQL + PostGIS | Реляційна СУБД із підтримкою просторових запитів |
| Клієнтська частина | React + TypeScript | Компонентний інтерфейс зі статичною типізацією |
| Клієнтська частина | MapLibre GL | Інтерактивна карта засобами WebGL |
| Клієнтська частина | Nivo | Діаграми ваг, чутливості та порівняння профілів |
| Клієнтська частина | TanStack Query | Взаємодія з REST API і кешування запитів |
| Тестування | pytest, Vitest, Playwright | Юніт, компонентне та наскрізне тестування |

Склад засобів спирається на три ключові принципи проектування.

1. Математичне ядро виокремлено в ізольований пакет `mcdm`, що не залежить від веб-фреймворку та засобів доступу до бази даних і спирається лише на NumPy і SciPy. Така ізоляція дає змогу перевіряти ядро юніт-тестами з числовими еталонами з літератури незалежно від решти системи.
2. Картографічне відображення побудовано на бібліотеці MapLibre GL [@maplibre_gl_docs_2025], що не потребує комерційних ключів доступу до картографічних сервісів.
3. Специфікація OpenAPI, яку FastAPI [@fastapi_docs_2025] генерує автоматично зі схем Pydantic, слугує єдиним контрактом між серверною і клієнтською частинами, а типи TypeScript для клієнта породжуються з цієї ж специфікації, чим усувається розбіжність структур даних між рівнями.

Спроектовану в підрозділі 2.2 логічну схему реалізовано в PostgreSQL 16 з розширенням PostGIS, а версіонування схеми забезпечують міграції Alembic [@alembic_docs_2025]. Геопросторові атрибути локацій-кандидатів і наявних станцій зберігаються типом `GEOMETRY(Point, 4326)` у системі координат WGS-84 з індексом GiST [@postgis_docs_2025], що уможливлює запити за відстанню. Доступ до даних асинхронний, через SQLAlchemy і GeoAlchemy2.

Особливістю реалізації ядра є пряма відповідність між алгоритмами підрозділу 2.3 і векторизованим кодом без явних циклів за альтернативами. Ваги критеріїв обчислюються методом Buckley відповідно до алгоритму підрозділу 2.3.1 – середнє геометричне рядків нечіткої матриці, перехід до нечітких ваг і центроїдна дефазифікація з нормалізацією, що наведено у фрагменті коду 3.1.

Фрагмент коду 3.1. – Обчислення ваг критеріїв методом Buckley

```python
n = matrix.shape[0]
z = np.prod(matrix, axis=1) ** (1.0 / n)             # (1.7) середнє геометричне
s_l, s_m, s_u = z.sum(axis=0)
w = z * np.array([1.0 / s_u, 1.0 / s_m, 1.0 / s_l])  # (1.8) нечіткі ваги
centroid = w.mean(axis=1)                            # (1.9) дефазифікація
weights = centroid / centroid.sum()                  #       нормалізація
```

Перед поверненням вектора ваг алгоритм перевіряє узгодженість матриці попарних порівнянь. Перевірку виконує окрема функція на модальних (середніх) значеннях нечіткої матриці – узгодженість є властивістю суджень, незалежно від способу виведення ваг. Перевірку винесено у функцію ядра, а не в сервісний шар, щоб тест міг подавати неузгоджену матрицю і отримувати виняток без запуску API (фрагмент коду 3.2).

Фрагмент коду 3.2. – Перевірка узгодженості матриці попарних порівнянь

```python
modal = matrix[:, :, 1]                              # модальні значення TFN
col_sums = modal.sum(axis=0)
w_approx = (modal / col_sums).mean(axis=1)           # (1.2) наближений власний вектор
lambda_max = np.mean((modal @ w_approx) / w_approx)  # (1.2) головне власне значення λ_max
ci = (lambda_max - n) / (n - 1)                      # (1.2) індекс узгодженості CI
cr = ci / RI[n]                                       # (1.2) коефіцієнт узгодженості CR
if cr > 0.10:
    raise ValueError(f"CR={cr:.3f} > 0.10")
```

Ранжування локацій будує TOPSIS відповідно до алгоритму підрозділу 2.3.2, реалізованого засобами NumPy, що наведено у фрагменті коду 3.3.

Фрагмент коду 3.3. – Ранжування локацій методом TOPSIS

```python
r = vector_normalize(decision_matrix)                    # (1.10) нормалізація
v = r * weights                                          # (1.11) зважена матриця
v_pos = np.where(types == 1, v.max(axis=0), v.min(axis=0))  # (1.12) ідеальний розв'язок
v_neg = np.where(types == 1, v.min(axis=0), v.max(axis=0))  # (1.12) антиідеальний розв'язок
s_pos = np.sqrt(((v - v_pos) ** 2).sum(axis=1))          # (1.13) відстані
s_neg = np.sqrt(((v - v_neg) ** 2).sum(axis=1))
denom = s_pos + s_neg                                    # (1.14) коефіцієнт
scores = np.where(denom == 0.0, 0.5, s_neg / denom)      # близькості
ranking = np.argsort(scores)[::-1]                       # ранжування за спаданням
```

Умова `np.where(denom == 0.0, 0.5, …)` усуває ділення 0/0: воно виникає, коли всі альтернативи однакові за всіма критеріями – тоді відстані $S_i^+$ і $S_i^-$ обидві дорівнюють нулю. Повернення нейтрального значення 0,5 замість виключення гарантує, що функція завжди повертає коректний масив чисел із плаваючою комою, а Монте-Карло завершує всі $N$ ітерацій навіть якщо збурені ваги породжують вироджений вектор. Аналіз чутливості, реалізований за алгоритмом підрозділу 2.3.3, багаторазово збурює ваги і ренормалізує їх, на кожній ітерації перебудовує ранжування і накопичує частоти рангів, з яких обчислює індекс прийнятності рангів; генератор `np.random.default_rng(seed)` створюється локально, а не через глобальний стан `np.random`. Це робить функцію реентерабельною: паралельний виклик двох аналізів або повторний запуск тесту не змінює стан генератора іншого виклику. Фіксоване зерно забезпечує побітово відтворюваний результат, що необхідно для порівняння двох профілів ОПР на одній матриці рішень (фрагмент коду 3.4).

Фрагмент коду 3.4. – Аналіз чутливості ранжування методом Монте-Карло

```python
rng = np.random.default_rng(seed)
rank_freq = np.zeros((n_alt, n_alt), dtype=np.intp)  # лічильник рангів
for _ in range(n_simulations):
    eps = rng.uniform(-delta, delta, size=n_crit)    # (1.15) збурення ваг
    w_t = base_weights * (1.0 + eps)
    w_t = w_t / w_t.sum()                            # (1.16) ренормалізація
    _, ranking = scorer(decision_matrix, w_t, types)
    for pos, alt in enumerate(ranking):
        rank_freq[pos, alt] += 1
# (1.17) індекс прийнятності: частка ітерацій із рангом ≤ k
top_k = np.cumsum(rank_freq, axis=0)[k - 1] / n_simulations
```

Три модулі обчислювального ядра – `fahp.py`, `topsis.py` і `monte_carlo.py` – реалізують послідовні етапи загального алгоритму функціонування системи (підрозділ 2.3.4): вихід кожного модуля є входом наступного, а спільний вектор ваг $W$ забезпечує їх взаємоузгодженість. Ізоляція ядра від решти системи підтверджена юніт-тестами з числовими еталонами з літератури.

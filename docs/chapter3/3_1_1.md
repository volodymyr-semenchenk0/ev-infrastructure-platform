# 3. РЕАЛІЗАЦІЯ СИСТЕМИ ЗА ТЕМОЮ КУРСОВОЇ РОБОТИ

## 3.1. Реалізація інтерфейсу та функцій програмного продукту

### 3.1.1. Обґрунтування вибору технологічного стеку

Вибір технологічного стеку визначається вимогами підрозділу 1.3: картографічна візуалізація, виконання циклу FAHP–TOPSIS–MC за $|A|=12$, $|C|=10$, $N=10\,000$ у межах 5 с, україномовний інтерфейс і два профілі ОПР. Порівнювались три поширені у задачах СППР рішення: Streamlit, Flask і FastAPI з React/TypeScript (табл. 3.1).

Таблиця 3.1 – Порівняльний аналіз технологічних рішень для реалізації СППР

| Характеристика | Streamlit | Flask + Jinja2 | FastAPI + React / TypeScript |
|---|---|---|---|
| Архітектура | Монолітна, SSR | Монолітна, SSR | Клієнт-серверна REST |
| Картографічна підтримка | обмежена (folium/st-folium) | обмежена (folium) | повна (react-leaflet + PostGIS) |
| Просторові запити до СУБД | відсутні | потребують raw SQL | GeoAlchemy2 + PostGIS нативно |
| Продуктивність MC ($N=10\,000$) | синхронний Python, блокує UI | синхронний Python | асинхронний FastAPI (asyncio) |
| Підтримка двох профілів ОПР | потребує складного session state | потребує сесій | нативний JWT або сесія |
| Масштабованість до 1000+ локацій | погана (reruns всієї сторінки) | середня | висока (REST + горизонтальне масштабування) |
| Гнучкість UI | низька (компоненти Streamlit) | середня (Jinja2 шаблони) | висока (React-компоненти) |

*Примітка. SSR – Server-Side Rendering; PostGIS – просторове розширення PostgreSQL; asyncio – стандартний модуль Python для асинхронного вводу/виводу.*

Streamlit виключено через відсутність підтримки просторових типів і блокуючі обчислення при $N=10\,000$. Flask потребує шаблонного коду для асинхронних обчислень і REST-контракту з підрозділу 2.1.6. Обрано FastAPI з React/TypeScript: цей стек нативно реалізує трирівневу архітектуру з підрозділу 2.1.1, виконує цикл асинхронно і підтримує просторові типи через GeoAlchemy2/PostGIS. Деталі шарів – у підрозділах 3.1.2–3.1.4.

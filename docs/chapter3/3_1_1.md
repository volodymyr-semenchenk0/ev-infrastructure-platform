# 3. РЕАЛІЗАЦІЯ СИСТЕМИ ЗА ТЕМОЮ КУРСОВОЇ РОБОТИ

## 3.1. Реалізація інтерфейсу та функцій програмного продукту

### 3.1.1. Обґрунтування вибору технологічного стеку

Вибір технологічного стеку для реалізації СППР визначається вимогами підрозділу 1.3: картографічна візуалізація геоіндексованих локацій, виконання обчислювального циклу FAHP–TOPSIS–MC за $|A|=12$, $|C|=10$, $N=10\,000$ у межах 5 с, україномовний інтерфейс, підтримка двох профілів ОПР з різними векторами ваг. Для вибору середовища реалізації порівнювались три технологічних рішення, найбільш поширені у задачах СППР: Streamlit, Flask та FastAPI з React і TypeScript. Зведений аналіз наведено у таблиці 3.1.

Таблиця 3.1. — Порівняльний аналіз технологічних рішень для реалізації СППР

| Характеристика | Streamlit | Flask + Jinja2 | FastAPI + React / TypeScript |
|---|---|---|---|
| Архітектура | Монолітна, SSR | Монолітна, SSR | Клієнт-серверна REST |
| Картографічна підтримка | обмежена (folium/st-folium) | обмежена (folium) | повна (react-leaflet + PostGIS) |
| Просторові запити до СУБД | відсутні | потребують raw SQL | GeoAlchemy2 + PostGIS нативно |
| Продуктивність MC ($N=10\,000$) | синхронний Python, блокує UI | синхронний Python | асинхронний FastAPI (asyncio) |
| Підтримка двох профілів ОПР | потребує складного session state | потребує сесій | нативний JWT або сесія |
| Масштабованість до 1000+ локацій | погана (reruns всієї сторінки) | середня | висока (REST + горизонтальне масштабування) |
| Гнучкість UI | низька (компоненти Streamlit) | середня (Jinja2 шаблони) | висока (React-компоненти) |

*Примітка. SSR — Server-Side Rendering; PostGIS — просторове розширення PostgreSQL; asyncio — стандартний модуль Python для асинхронного вводу/виводу.*

Streamlit виключено через відсутність повноцінної підтримки просторових типів, обмежену гнучкість інтерфейсу і неприйнятну продуктивність за умов блокуючих обчислень при $N=10\,000$ ітерацій Монте-Карло. Flask потребує суттєвого шаблонного коду для реалізації асинхронних обчислень і REST-контракту, описаного у підрозділі 2.1.6. Рішенням обрано FastAPI у поєднанні з React і TypeScript: цей стек нативно реалізує трирівневу клієнт-серверну архітектуру, спроєктовану у підрозділі 2.1.1, забезпечує виконання повного обчислювального циклу асинхронно та підтримує просторові типи через GeoAlchemy2 і PostGIS. Деталі реалізації кожного шару описані у підрозділах 3.1.2–3.1.4.

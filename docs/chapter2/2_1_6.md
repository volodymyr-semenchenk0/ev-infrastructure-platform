### 2.1.6. Специфікація REST API

Інтеграційний контракт між клієнтською й серверною підсистемами реалізується в архітектурному стилі REST. Вибір зумовлений трьома міркуваннями: ресурсна модель природно відображає сутності предметної галузі (профілі ОПР, критерії, локації, обчислювальні запити та результати); бездержавна (stateless) природа REST дозволяє горизонтальне масштабування серверної підсистеми як передумову готовності до 1000+ локацій (підрозділ 1.3); JSON-подання ресурсів тестується ізольовано від клієнтського інтерфейсу.

Контракт API будується на двох принципах. **Ресурсна модель URL:** кожен URL ідентифікує ресурс або колекцію; дієслова операцій винесено в HTTP-методи (GET / POST / PUT / DELETE); похідні ресурси використовують ієрархічні шляхи (`/api/evaluations/{id}/sensitivity`). **Семантика категорій відповідей:** успішне отримання, успішне створення з посиланням на ресурс, помилка валідації формату, семантична помилка ($CR > 0{,}1$), відсутній ресурс. Специфікацію ендпоінтів наведено у Табл. 2.1.

Таблиця 2.1. — Специфікація основних ендпоінтів REST API системи

| Метод | Шлях | Призначення | Тіло запиту (ключові поля) | Тіло відповіді (ключові поля) |
|---|---|---|---|---|
| GET | `/api/profiles` | Перелік профілів ОПР | — | масив `{id, code, name, description}` |
| GET | `/api/profiles/{id}` | Деталі профілю з векторами ваг і збереженою матрицею | — | `{id, code, name, criteria: [{id, code, weight}], pairwiseMatrix}` |
| GET | `/api/criteria` | Повний словник критеріїв | — | масив `{id, code, name, unit, optimizationType, scale}` |
| GET | `/api/locations` | Реєстр локацій-кандидатів | — | масив `{id, name, address, district, latitude, longitude}` |
| POST | `/api/locations` | Створення локації (лише адміністратор) | `{name, address, district, latitude, longitude}` | `{id, …}` |
| POST | `/api/evaluations` | Запуск повного циклу FAHP→TOPSIS | `{profileId, pairwiseMatrix: [[(l,m,u)]]}` | `{evaluationId, weights, ranking: [{locationId, rank, closeness, sPlus, sMinus}], executionTimeMs}` |
| GET | `/api/evaluations/{id}` | Отримання збереженого результату | — | те саме, що у відповіді POST `/api/evaluations` |
| POST | `/api/evaluations/{id}/sensitivity` | Запуск аналізу чутливості Монте-Карло | `{iterations, perturbation}` (за замовчуванням: 10000 і 0,15) | `{stabilityMatrix: [[p_i(k)]], confidenceIntervals: {locationId: [low, high]}}` |
| GET | `/api/evaluations/{id}/comparison/{otherId}` | Порівняння двох ранжувань різних профілів | — | `{spearmanRho, pairwiseDifferences: [{locationId, rankA, rankB, delta}]}` |
| GET | `/api/evaluations/{id}/export?format={csv\|json}` | Експорт результатів | — | документ у обраному форматі |

Ендпоінт `POST /api/evaluations` є центральним: тіло запиту містить `{profileId, pairwiseMatrix}` з матрицею у форматі двовимірного масиву трійок $(l_{ij}, m_{ij}, u_{ij})$ розміру $m \times m$. Сервер виконує дворівневу валідацію: спочатку формату (розмірність, взаємна оберненість $\tilde{a}_{ji} = 1 / \tilde{a}_{ij}$, діапазон шкали Саати $[1/9; 9]$) — відповідь категорії «помилка валідації формату»; потім семантична перевірка $CR \leq 0{,}1$ усередині FAHP-сервісу — відповідь категорії «семантична помилка» з рекомендацією переглянути судження. За умови успіху — відповідь категорії «успішне створення» з вектором ваг, повним ранжуванням ($C_i^*$, $S_i^+$, $S_i^-$ для всіх 12 локацій) і часом виконання.

Усі ендпоінти Табл. 2.1 є stateless: жодна операція не зберігає стану між запитами, що дозволяє розгортати серверну підсистему у багатоекземплярному режимі без узгодження сесій.

Інформаційне забезпечення системи — концептуальну й логічну моделі бази даних та потоки зовнішніх вхідних даних — описано у наступному підрозділі.

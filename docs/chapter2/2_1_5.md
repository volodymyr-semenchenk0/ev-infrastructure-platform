### 2.1.5. Діаграма комунікації

Діаграма комунікації – альтернативне подання сценарію 2.1.4 без часової осі з акцентом на топології зв'язків між об'єктами (чим менше прямих зв'язків, тим вища модульність). Діаграму наведено на рис. 2.5.

![Діаграма комунікації для сценарію «Обчислити ранжування локацій». Учасники: ОПР, Client, RESTController, OrchestrationService, FAHPCalculator, TOPSISRanker, DataRepository, СУБД. Нумеровані повідомлення: 1 – ОПР→Client; 2 – Client→RESTController (POST /api/evaluations); 3 – RESTController→Orch (executeFullCycle); 3.1 – Orch→Repo (findCriteriaByProfile); 3.2 – Orch→FAHP; 3.3 – Orch→Repo (findLocationsWithMetrics); 3.4 – Orch→TOP (computeRanking); 3.5 – Orch→Repo (saveEvaluation); 4 – RESTController→Client (відповідь); 5 – Client→ОПР (відображення)](images/fig_2_5_communication_diagram.png)

Рис. 2.5. Діаграма комунікації для сценарію «Обчислити ранжування локацій»

```plantuml
@startuml
skinparam shadowing false
skinparam defaultFontName "Times New Roman"
skinparam defaultFontSize 11
skinparam object {
    BackgroundColor #f5f5f5
    BorderColor #333333
}
skinparam database {
    BackgroundColor #eaeaea
    BorderColor #333333
}

object ":ОПР" as User
object ":Client" as Client
object ":RESTController" as Ctrl
object ":OrchestrationService" as Orch
object ":FAHPCalculator" as FAHP
object ":TOPSISRanker" as TOP
object ":DataRepository" as Repo
database ":СУБД" as DB

User --> Client : 1: запуск обчислень
Client --> Ctrl : 2: POST /api/evaluations
Ctrl --> Orch : 3: executeFullCycle(profileId, matrix)
Orch --> Repo : 3.1: findCriteriaByProfile(profileId)
Repo --> DB : 3.1.1: SELECT критерії
Orch --> FAHP : 3.2: computeWeights(matrix)
Orch --> Repo : 3.3: findLocationsWithMetrics()
Repo --> DB : 3.3.1: SELECT локації + значення
Orch --> TOP : 3.4: computeRanking(X, W)
Orch --> Repo : 3.5: saveEvaluation(run)
Repo --> DB : 3.5.1: INSERT обчислювальний запит
Ctrl --> Client : 4: успішна відповідь
Client --> User : 5: відобразити ранжування

@enduml
```

Топологія виявляє дві властивості: `FAHPCalculator` і `TOPSISRanker` не мають прямого зв'язку – обидва клієнтуються лише через `OrchestrationService` (єдина відповідальність, ізольована тестованість); усі звернення до `DataRepository` централізовані – жоден сервіс не звертається до сховища напряму, що критично для внутрішнього циклу МК ($N = 10\,000$ викликів TOPSIS).

### 2.1.5. Діаграма комунікації

Діаграма комунікації — альтернативне представлення того самого сценарію «Обчислити ранжування локацій», що і Sequence Diagram (підрозділ 2.1.4), але без часової осі: акцент зроблено на топології зв'язків між об'єктами. Чим менше прямих зв'язків — тим вищий рівень модульності. Діаграму наведено на рис. 2.5.

![Діаграма комунікації для сценарію «Обчислити ранжування локацій». Учасники: ОПР, Client, RESTController, OrchestrationService, FAHPCalculator, TOPSISRanker, DataRepository, СУБД. Зв'язки між учасниками з нумерованими повідомленнями: 1 — ОПР→Client (запуск обчислень); 2 — Client→RESTController (POST /api/evaluations); 3 — RESTController→OrchestrationService (executeFullCycle); 3.1 — OrchestrationService→DataRepository (findCriteriaByProfile); 3.1.1 — DataRepository→СУБД (SELECT критерії); 3.2 — OrchestrationService→FAHPCalculator (computeWeights); 3.3 — OrchestrationService→DataRepository (findLocationsWithMetrics); 3.3.1 — DataRepository→СУБД (SELECT локації); 3.4 — OrchestrationService→TOPSISRanker (computeRanking); 3.5 — OrchestrationService→DataRepository (saveEvaluation); 3.5.1 — DataRepository→СУБД (INSERT); 4 — RESTController→Client (успішна відповідь з ресурсним поданням); 5 — Client→ОПР (відображення)](images/fig_2_5_communication_diagram.png)

Рис. 2.5. Діаграма комунікації для сценарію «Обчислити ранжування локацій»

```plantuml
@startuml
skinparam shadowing false
skinparam defaultFontName "Times New Roman"
skinparam defaultFontSize 11
skinparam object {
    BackgroundColor #f5f5f5
    BorderColor #333333
}
skinparam database {
    BackgroundColor #eaeaea
    BorderColor #333333
}

object ":ОПР" as User
object ":Client" as Client
object ":RESTController" as Ctrl
object ":OrchestrationService" as Orch
object ":FAHPCalculator" as FAHP
object ":TOPSISRanker" as TOP
object ":DataRepository" as Repo
database ":СУБД" as DB

User --> Client : 1: запуск обчислень
Client --> Ctrl : 2: POST /api/evaluations
Ctrl --> Orch : 3: executeFullCycle(profileId, matrix)
Orch --> Repo : 3.1: findCriteriaByProfile(profileId)
Repo --> DB : 3.1.1: SELECT критерії
Orch --> FAHP : 3.2: computeWeights(matrix)
Orch --> Repo : 3.3: findLocationsWithMetrics()
Repo --> DB : 3.3.1: SELECT локації + значення
Orch --> TOP : 3.4: computeRanking(X, W)
Orch --> Repo : 3.5: saveEvaluation(run)
Repo --> DB : 3.5.1: INSERT обчислювальний запит
Ctrl --> Client : 4: успішна відповідь
Client --> User : 5: відобразити ранжування

@enduml
```

Топологія виявляє дві ключові властивості: `FAHPCalculator` і `TOPSISRanker` не мають прямого зв'язку між собою — обидва клієнтуються виключно `OrchestrationService` (принцип єдиної відповідальності, ізольована тестованість); усі звернення до `DataRepository` централізовані в `OrchestrationService` — жоден обчислювальний сервіс не звертається до сховища напряму, що критично для внутрішнього циклу Монте-Карло ($N = 10\,000$ викликів `TOPSISRanker`).

Інтеграційний контракт між клієнтською та серверною підсистемами — структуру REST API — формалізовано у наступному підрозділі.
